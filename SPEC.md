# SPEC.md — 功能規格書

本文件是 Many Money 的功能規格書，以「想透過 AI 輔助記帳達成開源節流」的使用者視角定義系統應具備的功能。
優先級判讀規則、標記方式請見 [AGENTS.md](AGENTS.md) 第 5 章。

## 產品定位

一個 AI 輔助記帳軟體，核心目的是**幫助使用者開源節流**。包含四大支柱：

1. **完整記帳功能** — 收支記錄、分類、帳本管理
2. **家庭共享** — 多人共用帳本，共同管理家庭財務
3. **遊戲化（Duolingo 式情勒）** — 用連續記錄、提醒、激勵機制養成記帳習慣
4. **AI 金流分析** — 分析消費模式，提出具體的開源節流建議

---

## P0 — 核心記帳功能（缺少會讓產品無法使用）

### P0-1: 使用者帳號與登入

**使用者故事**：作為一個使用者，我想要註冊並登入帳號，這樣我的記帳資料才能被安全地保存與同步。

**驗收標準**：
- [ ] 支援 Email + 密碼註冊/登入
- [ ] 支援至少一種 OAuth（Google）快速登入
- [ ] 登入狀態可跨裝置（網頁 / app）同步
- [ ] 密碼/Token 等敏感資訊不可明文儲存

**技術方案**：
- 後端：Auth 服務（建議 NextAuth.js / Auth.js，搭配 JWT or session）
- 資料庫：User 資料表

**優先級理由**：沒有帳號系統，無法做家庭共享、跨裝置同步，是一切功能的地基。

---

### P0-2: 交易記錄 CRUD

**使用者故事**：作為一個使用者，我想要快速新增/編輯/刪除一筆收入或支出，這樣我才能完整記錄金流。

**驗收標準**：
- [ ] 可新增交易：金額、類型（收入/支出/轉帳）、分類、日期、帳本、備註、（選填）照片/憑證
- [ ] 可編輯、刪除既有交易
- [ ] 列表可依日期、分類、帳本篩選與排序
- [ ] 金額需做防呆（不可為負數除非是轉帳語意、不可為 NaN）

**技術方案**：
- 前端：交易表單、列表元件（utils 層處理金額/日期格式化與驗證）
- 後端：Transaction CRUD API
- 資料庫：Transaction 資料表（userId, ledgerId, amount, type, categoryId, date, note, attachmentUrl）

**優先級理由**：核心記帳行為，產品的最小可行功能。

---

### P0-3: 分類管理

**使用者故事**：作為一個使用者，我想要將交易歸類到自訂或預設分類，這樣我才能看懂錢花在哪裡。

**驗收標準**：
- [ ] 提供常用預設分類（餐飲、交通、housing、娛樂、薪資等）
- [ ] 可新增/編輯/刪除自訂分類，並指定圖示/顏色
- [ ] 刪除分類時，既有交易需妥善處理（轉為「未分類」而非報錯或遺失資料）

**技術方案**：
- 資料庫：Category 資料表（userId/ledgerId, name, icon, color, type）

**優先級理由**：沒有分類，後續的統計與 AI 分析都無法進行。

---

### P0-4: 帳本（Ledger）管理

**使用者故事**：作為一個使用者，我想要建立多個帳本（如「個人」「家庭」「旅遊基金」），這樣不同性質的金流可以分開管理。

**驗收標準**：
- [ ] 可建立/編輯/刪除帳本
- [ ] 每筆交易必須歸屬於一個帳本
- [ ] 預設建立一個「個人帳本」

**技術方案**：
- 資料庫：Ledger 資料表（ownerId, name, currency, type: personal/shared）

**優先級理由**：是家庭共享功能（P1）的前置依賴。

---

### P0-5: 基礎報表（收支總覽）

**使用者故事**：作為一個使用者，我想要看到本月/本週的收支總覽，這樣我才能掌握現在的財務狀況。

**驗收標準**：
- [ ] 顯示指定時間區間的總收入、總支出、淨額
- [ ] 依分類顯示支出占比（圖表）
- [ ] 可切換時間區間（本週/本月/自訂區間）

**技術方案**：
- 前端：圖表元件（如 Recharts）
- 後端：聚合查詢 API（純計算邏輯放 utils，並寫測試）

**優先級理由**：使用者需要立即看到記帳的回饋，否則沒有動力持續使用。

---

## P1 — 家庭共享

### P1-1: 邀請家庭成員共用帳本

**使用者故事**：作為一個使用者，我想要邀請家人加入我的帳本，這樣我們可以一起記錄家庭收支。

**驗收標準**：
- [ ] 可透過邀請連結或 email 邀請其他使用者加入共享帳本
- [ ] 被邀請者需明確同意才能加入
- [ ] 帳本擁有者可移除成員、轉移擁有權

**技術方案**：
- 資料庫：LedgerMember 資料表（ledgerId, userId, role: owner/editor/viewer）
- 後端：邀請 token 產生與驗證流程

**優先級理由**：家庭共享是產品核心差異化功能之一。

---

### P1-2: 共享帳本的權限控管

**使用者故事**：作為帳本擁有者，我想要設定成員權限（可編輯 / 僅檢視），這樣可以避免誤刪或誤改家人的記錄。

**驗收標準**：
- [ ] 至少支援 owner / editor / viewer 三種角色
- [ ] viewer 無法新增/編輯/刪除交易
- [ ] 所有寫入 API 都需做權限檢查（不可僅靠前端隱藏按鈕）

**技術方案**：
- 後端：權限檢查 middleware，集中管理，不可散落在各 handler

**優先級理由**：多人共用資料必須有權限邊界，否則家庭關係會因誤操作受影響。

---

### P1-3: 家庭帳本的成員貢獻視圖

**使用者故事**：作為家庭成員，我想要看到誰記了哪些支出，這樣家人可以互相提醒、核對。

**驗收標準**：
- [ ] 交易列表可顯示記錄者
- [ ] 可依成員篩選交易
- [ ] （可選）可看到每位成員本月記錄筆數，作為遊戲化的延伸

**技術方案**：
- 前端：交易列表加上 recordedBy 顯示與篩選

**優先級理由**：增加共享帳本的透明度與互動感，是遊戲化（P2）的銜接點。

---

## P2 — 遊戲化（Duolingo 式情勒機制）

### P2-1: 連續記帳天數（Streak）

**使用者故事**：作為一個使用者，我想要看到自己連續記帳的天數，這樣我會有動力每天持續記錄。

**驗收標準**：
- [ ] 每天至少記一筆交易才算「當日完成」
- [ ] 中斷一天，streak 歸零，並有明確視覺提示
- [ ] 首頁顯著顯示目前 streak 數字

**技術方案**：
- 資料庫：UserStreak 資料表（userId, currentStreak, longestStreak, lastRecordedDate）
- 後端：每次新增交易時更新 streak 的純函式邏輯（需覆蓋時區邊界測試）

**優先級理由**：Streak 是 Duolingo 式機制的核心，直接驅動留存率。

---

### P2-2: 情勒提醒通知

**使用者故事**：作為一個使用者，我想要在快要中斷 streak 或快超支時收到提醒通知，這樣我會被「情勒」去採取行動。

**驗收標準**：
- [ ] 當日尚未記帳且接近一天結束時，推播提醒（app push / email）
- [ ] 月度預算即將超支時提醒
- [ ] 文案語氣需可調（俏皮/嚴肅），但不可造成使用者反感而棄用

**技術方案**：
- 後端：排程任務（cron）+ 推播服務（FCM / web push）
- 前端：app 需整合推播 SDK（Capacitor push plugin）

**優先級理由**：通知機制是遊戲化情勒效果能否落地的關鵧，但需與 app 包裝（Capacitor）整合，技術上依賴度較高。

---

### P2-3: 預算目標與進度條

**使用者故事**：作為一個使用者，我想要為分類設定每月預算，這樣我可以直觀看到「還能花多少」。

**驗收標準**：
- [ ] 可針對分類或總體設定月度預算
- [ ] 首頁/分類頁顯示已花費 / 預算 的進度條
- [ ] 超出預算時有明確的視覺警示（變色、圖示）

**技術方案**：
- 資料庫：Budget 資料表（userId/ledgerId, categoryId, amount, period）
- 前端：純函式計算花費占比，附完整邊界測試（預算為 0、超支、無資料）

**優先級理由**：是「節流」目標的直接體現，也是情勒通知（P2-2）的資料來源。

---

### P2-4: 成就 / 徽章系統

**使用者故事**：作為一個使用者，我想要在達成特定目標（如連續記帳 30 天、達成節流目標）時獲得成就徽章，這樣記帳會更有成就感。

**驗收標準**：
- [ ] 至少設計 5-10 個初版成就（streak 里程碑、首次設定預算、首次完成 AI 建議等）
- [ ] 成就達成時有明確的慶祝動畫/提示
- [ ] 個人頁面可瀏覽已獲得 / 未獲得的成就

**技術方案**：
- 資料庫：Achievement、UserAchievement 資料表
- 後端：成就判定邏輯集中在獨立的純函式模組，方便擴充新成就

**優先級理由**：屬於黏著度功能，可在核心功能（P0/P1）穩定後排入。

---

## P3 — AI 金流分析

### P3-1: 消費模式分析報告

**使用者故事**：作為一個使用者，我想要 AI 幫我分析最近的消費紀錄，這樣我可以知道錢都花到哪裡、哪裡可以改善。

**驗收標準**：
- [ ] 可產生週期性（月度）分析報告：消費分佈、與上月比較、異常支出標記
- [ ] 報告需以使用者能懂的自然語言呈現，不只是圖表
- [ ] 分析結果需基於使用者實際資料，不可產生與資料矛盾的結論（防止幻覺）

**技術方案**：
- 後端：先用純函式做資料聚合（純計算，可測試），再將聚合後的結構化資料丟給 LLM 生成敘述，**不要讓 LLM 直接做數值運算**
- 需明確記錄 LLM prompt 與輸出版本，方便除錯與迴歸測試

**優先級理由**：AI 分析是產品的賦能型功能，依賴 P0 的資料完整度，因此排在記帳基礎之後。

---

### P3-2: 個人化節流建議

**使用者故事**：作為一個使用者，我想要 AI 根據我的消費習慣，提出具體可執行的節流建議，這樣我才能真正改善財務狀況。

**驗收標準**：
- [ ] 建議必須具體（例如「外食佔比過高，建議...」），不可是空泛的通用建議
- [ ] 每條建議需附上對應的數據依據（如「本月外食花費 $X，較上月增加 Y%」）
- [ ] 使用者可標記建議「已採納」「不適用」，作為後續模型調整的回饋資料

**技術方案**：
- 後端：建議生成邏輯需可重現（同樣輸入應產生一致風格的建議），考慮加上 cache 避免重複呼叫 LLM 浪費成本
- 資料庫：Suggestion 資料表（userId, content, basedOnData, status: pending/accepted/dismissed）

**優先級理由**：直接對應產品「開源節流」的最終目的，是差異化賣點，但技術風險與成本較高（LLM 呼叫費用、品質控管），排在核心功能穩定後。

---

### P3-3: 開源（增加收入）建議

**使用者故事**：作為一個使用者，我想要 AI 不只幫我省錢，也能根據我的財務狀況提出增加收入的方向建議。

**驗收標準**：
- [ ] 至少能根據既有收入結構（如收入來源單一）提出風險提示
- [ ] 建議內容需明確標示為「一般性參考建議」，不可構成具體投資/理財建議的誤導（法律風險）

**技術方案**：
- 與 P3-2 共用建議生成基礎設施，但需獨立 prompt 與免責聲明

**優先級理由**：「開源」相對「節流」更難量化且有法律風險（理財建議邊界），先以保守、提示性的內容上線。

---

## P4 — 進階 / 待評估

### P4-1: 跨裝置 App（Capacitor 包裝）

**使用者故事**：作為一個使用者，我想要在手機上用 app 的方式記帳，這樣比開瀏覽器更方便、也能收到推播通知。

**驗收標準**：
- [ ] 使用 Capacitor 包裝 Next.js 網頁為 iOS / Android app
- [ ] App 內可正常使用推播通知（依賴 P2-2）
- [ ] App store 上架基本素材（icon、screenshot、隱私政策頁）

**技術方案**：
- Capacitor + 既有 Next.js 專案，視效能需求評估是否需要拆出純前端靜態匯出

**優先級理由**：依賴核心網頁功能先穩定，且上架審核需要額外時間規劃，列為中後期目標。

---

### P4-2: 帳單/憑證自動辨識（OCR）

**使用者故事**：作為一個使用者，我想要拍照上傳收據，AI 自動辨識金額與分類，這樣我不用手動輸入。

**驗收標準**：
- [ ] 拍照後可自動填入金額、商家、建議分類
- [ ] 辨識錯誤時使用者可手動修正，修正結果可作為未來改善依據

**技術方案**：
- 需評估 OCR 服務成本與準確率（雲端 Vision API vs 自建模型）

**優先級理由**：屬於體驗加分功能，非必要路徑，且有外部服務成本，待核心功能穩定後評估。

---

## P5 — 實驗性 / 需外部資源

### P5-1: 銀行/信用卡自動同步

**使用者故事**：作為一個使用者，我想要自動同步銀行/信用卡交易紀錄，這樣不需要手動記帳。

**驗收標準**：（待評估，依賴第三方金融資料整合服務，如 Plaid 等同類台灣本地服務的可行性）

**技術方案**：
- 需研究台灣銀行端是否有對應的開放銀行 API 或第三方整合服務，目前無已知免費穩定方案

**優先級理由**：高度依賴外部資源與法規（金融資料存取），記錄需求但暫不投入開發。

## 附錄：開發順序建議

```
Phase 1（MVP）：P0-1 ~ P0-5（帳號、交易 CRUD、分類、帳本、基礎報表）
Phase 2：P1-1 ~ P1-3（家庭共享）
Phase 3：P2-1 ~ P2-3（streak、提醒、預算進度）
Phase 4：P3-1 ~ P3-2（AI 分析、節流建議）
Phase 5：P2-4、P3-3、P4-1（成就系統、開源建議、App 包裝）
Phase 6（視評估結果）：P4-2、P5-1
```
